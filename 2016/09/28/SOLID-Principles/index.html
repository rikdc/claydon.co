SOLID Principles

2016-09-28

Well, I completely bungled a job interview by failing to describe what S.O.L.I.D is, even though it’s part of my daily programming life and I’ve even asked programers the same question when I’ve interviewed them. So for my punishment, here is a little revision.

SOLID stands for five basic principles for developing good quality software architecture. It’s five principles are:

Single Responsibility
Open closed
LLiskov substitution
Interface segreation
Dependency inversion

Single Responsibility

A class should have one single responsibility, and this should be encapsulated by the class. There should only be one reason for creating the class.

When I come across classes that violate this, it’s easy to tell. They are the classes with named like ‘Manager’ and have thousands of lines of code. Refactor that up into smaller, cleaner classes for better quality code.

Open Closed

Application code should be open for extension, but closed for modification. Simply put, new functionality should be added through new classes and have minimal requirement to modify existing functionality.

How would I do this? Through the use of interfaces mainly, but there are other approaches such as subclasses to help abstract any implementation details.

Liskov Substitution

This principle states that any derived class should be perfectly substitutable for its base class.

Interface Segregation

No client should be forced to implement methods which it does not use, and contracts should be broken down into thin ones.

Similar to SRP, this is about creating smaller interfaces to keep your code decoupled.

Dependency Inversion

Your high level class should not depend on your low level classes, this is about abstractions and keeping the layers of your software decoupled.

I started using this principle when I really started seeing the light with unit tests, and using dependency injection became central to a lot of the code I was writing. Say I have a service class responsible for updating stock quantities and the new stock quantities are to be persisted to a database. There’s a calcuation operation there and a persistence operation. The persistence should be abstracted into its own class, and the calculation class would have the persistence class injected into it.

I’m not the most articulate of people, but maybe next time I get asked this question in an interview I’ll be able to provide a slightly more succinct answer than before.
